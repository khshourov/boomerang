# Task 5.3: CLI Integration for Task Listing

Implement the `task list` and `task get <id>` commands in the `boomtool` CLI, and update the `BoomerangClient` to support these management operations.

## Sub-tasks

1.  **Update `BoomerangClient`:**
    *   Add `listTasks(ListTasksRequest request)` method to return `ListTasksResponse`.
    *   Add `getTask(String taskId)` method to return `GetTaskResponse`.
    *   Ensure these methods use the existing `sendRequest` mechanism for session-aware communication.

2.  **Implement `TaskListCommand`:**
    *   Create `io.boomerang.cli.TaskListCommand` extending `BaseCommand`.
    *   Add CLI options:
        *   `--client-id`: Filter by client (Admin only).
        *   `--after`: Filter tasks scheduled after this interval from now (e.g., `5m`, `1h`).
        *   `--before`: Filter tasks scheduled before this interval from now (e.g., `2h`, `1d`).
        *   `--recurring`: Filter for recurring tasks (`true`/`false`).
        *   `--limit`: Maximum number of tasks to display (default 20).
        *   `--all`: If set, automatically fetch all pages using `next_token`.
    *   Format the output as a simple table using `printf`, showing `Task ID`, `Client ID`, `Expiration (ms)`, and `Recurring`.

3.  **Implement `TaskGetCommand`:**
    *   Create `io.boomerang.cli.TaskGetCommand` extending `BaseCommand`.
    *   Accept `taskId` as a positional parameter.
    *   Display full details of the task, including the binary payload decoded as a UTF-8 string.

4.  **Register Commands in `BoomTool`:**
    *   Add `TaskListCommand.class` and `TaskGetCommand.class` to the subcommands of `BoomTool.TaskCommand`.

5.  **Verification & Testing:**
    *   Add unit tests for `TaskListCommand` and `TaskGetCommand` using mock `BoomerangClient`.
    *   Verify pagination logic (the `--all` flag or manual `next_token` usage).
    *   Verify interval-to-epoch conversion logic.
    *   Verify interactive mode support for these new commands.

## Clarification Questions

1.  **Interval Parsing:** Should we implement a custom duration parser for strings like `1h30m`? (Recommended: yes, similar to Prometheus or Go duration strings).
2.  **Table Headers:** Are there any specific columns you'd like to see besides the ones mentioned?
3.  **Empty Payload:** How should we handle payloads that are not valid UTF-8? (Proposed: Fallback to hex or show "[Binary Data]").
